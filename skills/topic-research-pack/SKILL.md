# Topic Research Pack

## 目标

帮助用户围绕一个新主题生成一套「双层研究资料包」：

1. **快速理解版 Markdown**：适合 5-10 分钟快速建立基本认知。
2. **深度研究版 PDF**：10000 中文字以上，适合认真学习、写文章、做选题、做技术判断或辅助决策。

这不是普通科普回答。执行本 Skill 时，要产出可保存、可阅读、可转发的研究文件，并尽量做到信息密度高、结构清晰、来源可追溯。

## 必须产出的文件

每次执行都要生成两份最终文件：

```text
[主题]-快速理解.md
[主题]-深度研究报告.pdf
```

建议同时保留 PDF 的 Markdown 源稿，便于用户后续编辑：

```text
[主题]-深度研究报告.md
```

命名要求：

- 文件名应使用用户主题的中文或英文名称。
- 文件名避免特殊符号；必要时用短横线或下划线替代。
- 如果主题名称过长，提炼为 2-6 个关键词。

## 适用场景

当用户提出以下类型请求时，使用本 Skill：

- “帮我研究 MCP，生成快速理解和详细 PDF”
- “系统了解 RAG，要求有学术根源、技术路线和产业落地”
- “帮我做一份 AI Agent 的深度研究报告”
- “拆解某个技术的发展历程、竞品和未来趋势”
- “我想快速理解一个新知识点，并要一份万字以上 PDF”
- “给我一份带来源的研究资料包”

不适用场景：

- 用户只要一句话解释。
- 用户只要简单翻译或摘要。
- 用户明确只要聊天式讲解，不需要文件。
- 用户要求写代码、修 bug、处理表格等非研究报告任务。

## 总体工作流

按以下流程执行：

1. **识别主题**：确定用户要研究的主题，以及它属于技术、产品、行业、理论、工具、趋势还是商业模式。
2. **判断范围**：如果主题过大，主动收敛范围；如果用户已经指定重点，优先满足指定重点。
3. **搜集信息来源**：对当前性强、涉及产品/公司/论文/竞品/产业落地的主题，必须联网搜索。
4. **优先读取权威来源**：优先官方文档、论文、GitHub 仓库、公司工程博客、行业报告、新闻报道。
5. **提取研究要素**：整理起源、演进、学术根源、核心概念、技术机制、竞争范式、技术路线、产业落地、竞品、案例、风险、趋势。
6. **生成快速理解版 Markdown**：控制在 1500-3000 中文字左右，适合快速阅读。
7. **生成深度研究版 Markdown 源稿**：内容应达到 10000 中文字以上或等价深度。
8. **检查完整性**：确保深度报告包含必备章节、来源链接和关键表格。
9. **转换为 PDF**：生成排版清晰的 PDF。
10. **返回文件路径**：告诉用户两份最终文件的位置；如保留源稿，也一并说明。

## 信息来源要求

对需要时效性或事实支撑的主题，必须使用信息来源，不要只凭常识生成。

优先级如下：

1. 官方文档、官网、标准规范
2. 学术论文、arXiv、会议论文、教材资料
3. GitHub 仓库、开源项目 README、Release Notes
4. 公司博客、工程博客、技术白皮书
5. 行业报告、咨询机构报告、市场研究
6. 新闻报道、访谈、发布会材料
7. 高质量技术博客、社区讨论

引用要求：

- 深度报告必须包含「参考资料与信息来源」章节。
- 关键事实、时间节点、论文、产品、公司、指标等应尽量给出来源。
- 如果某些判断属于分析或推测，要明确标注为“分析判断”或“趋势判断”。
- 不要编造来源、论文、公司或链接。

## 文件一：快速理解版 Markdown

### 定位

让用户在 5-10 分钟内快速建立认知，知道这个主题是什么、为什么重要、发展到哪一步、和谁竞争、有什么用。

### 长度

建议 1500-3000 中文字。不要写成大而全报告。

### 结构

快速理解版 Markdown 必须使用以下结构：

```markdown
# [主题] 快速理解

## 1. 一句话理解

用一句通俗但准确的话说明它是什么。

## 2. 它解决什么问题？

说明它为什么出现，替代或补充了什么旧方式，解决了什么痛点。

## 3. 核心要点

用 5-8 条 bullet 总结最重要的信息。

## 4. 发展脉络速览

用简短时间线说明关键节点。

| 时间 | 事件 | 意义 |
|---|---|---|

## 5. 和谁竞争？

说明它面对的旧范式、替代方案、相邻技术或类似竞品。

## 6. 现在用在哪里？

列出主要产业落地场景和代表案例。

## 7. 一句话判断

总结它当前的价值、局限和未来趋势。

## 8. 主要信息来源

列出 5-10 个最重要的信息来源链接。
```

### 风格

- 直接、清楚、少废话。
- 适合发给不熟悉该主题的人快速阅读。
- 不要堆术语；出现关键术语时要解释。
- 可以使用表格，但不要过度展开。

## 文件二：深度研究版 PDF

### 定位

面向需要深入研究的人，适合用于：

- 写公众号文章
- 做技术选型
- 做行业研究
- 准备分享或汇报
- 快速补齐一个新领域认知

### 长度硬性要求

深度研究报告应达到 **10000 中文字以上**，或英文主题下达到等价信息密度。

如果初稿不足，应优先扩展以下章节：

- 起源与问题背景
- 发展历程与关键节点
- 学术根源与理论基础
- 竞争范式分析
- 技术路线对比
- 产业落地与商业化场景
- 类似竞品 / 代表玩家
- 典型案例分析
- 风险、争议与未来趋势

### 深度报告结构

深度研究 PDF 的 Markdown 源稿必须使用以下结构：

```markdown
# [主题] 深度研究报告

## 0. 信息来源说明

说明本报告主要基于哪些类型的信息来源，包括官方文档、论文、GitHub、公司博客、行业报告、新闻或其他资料。

## 1. 执行摘要

用 8-12 条 bullet 总结全文核心结论：

- 它是什么
- 为什么出现
- 当前发展阶段
- 核心价值
- 关键技术变化
- 竞争范式
- 产业落地情况
- 代表玩家
- 主要风险
- 未来趋势

## 2. 一句话理解与基本定义

包括：

- 一句话理解
- 通俗解释
- 专业定义
- 适用边界
- 它不是什么

## 3. 起源与问题背景

回答：

- 它最早为了解决什么问题？
- 旧方案是什么？
- 旧方案有什么瓶颈？
- 为什么这个问题在当下变得重要？
- 哪些技术、市场或产业变化推动了它出现？

## 4. 发展历程与关键节点

优先使用时间线：

| 时间 | 阶段/事件 | 代表人物/机构/产品 | 影响 | 来源 |
|---|---|---|---|---|

如果资料不足，用阶段式：

- 概念萌芽期
- 技术探索期
- 工程化落地期
- 生态竞争期
- 标准化/平台化期

## 5. 学术根源与理论基础

包括：

- 相关学科来源
- 基础理论
- 关键论文
- 代表性算法、模型或方法
- 学术思想如何转化为工程实践
- 当前学术界仍在研究的问题

## 6. 核心概念与知识地图

用表格解释关键概念：

| 概念 | 通俗解释 | 专业含义 | 为什么重要 |
|---|---|---|---|

再给出文字版知识地图：

```text
理论基础
  └── 核心机制
        └── 工程实现
              └── 应用场景
                    └── 商业生态
```

## 7. 工作原理 / 技术机制

讲清楚它如何运行：

```text
输入 → 处理流程 → 输出 → 反馈/优化
```

技术主题需要补充：

- 系统架构
- 数据流
- 关键组件
- 工程实现方式
- 性能瓶颈
- 安全与可靠性问题

## 8. 竞争范式分析

回答：

- 它属于什么新范式？
- 它挑战了哪些旧范式？
- 同一问题还有哪些不同解决路径？
- 各范式的核心假设是什么？

使用表格：

| 范式 | 核心假设 | 代表方案 | 优势 | 局限 |
|---|---|---|---|---|

## 9. 技术路线对比

使用表格：

| 技术路线 | 基本原理 | 适用场景 | 优点 | 缺点 | 成熟度 |
|---|---|---|---|---|---|

并判断：

- 哪些路线已经成熟？
- 哪些路线还在早期？
- 哪些路线可能只是过渡方案？
- 哪些路线更可能成为主流？

## 10. 产业落地与商业化场景

使用表格：

| 行业/场景 | 典型需求 | 解决方式 | 代表公司/产品 | 落地难点 |
|---|---|---|---|---|

重点分析：

- 哪些行业最先采用
- 解决了什么真实业务问题
- 商业模式是什么
- 采购、部署、使用门槛是什么
- 当前落地处于什么阶段

## 11. 类似竞品 / 代表玩家

如果是产品、工具、平台、生态，列竞品：

| 产品/项目 | 定位 | 核心能力 | 差异点 | 适合场景 | 官网/来源 |
|---|---|---|---|---|---|

并分组：

- 第一梯队
- 第二梯队
- 开源代表
- 商业代表
- 新兴玩家

如果不是产品型主题，则改为：

- 相邻概念
- 替代方案
- 相关技术

## 12. 典型案例分析

至少 3 个案例：

### 案例一：[公司/产品/项目]

- 背景
- 使用方式
- 解决的问题
- 效果
- 局限
- 来源

## 13. 关键争议、风险与限制

包括：

- 技术争议
- 商业争议
- 安全风险
- 成本问题
- 生态锁定
- 监管/伦理问题
- 数据隐私问题
- 长期可持续性

## 14. 未来趋势判断

分为：

### 短期趋势：1 年内
### 中期趋势：1-3 年
### 长期趋势：3-5 年
### 最大不确定性

注意区分事实与趋势判断。

## 15. 学习路径

分三层：

### 入门阶段
### 进阶阶段
### 实践阶段

每阶段说明：

- 学什么
- 看什么资料
- 做什么练习
- 如何判断自己学会了

## 16. 自测问题

至少 8-10 个问题，检查用户是否真正理解：

1. 它解决的核心问题是什么？
2. 它为什么在这个时间点兴起？
3. 它和旧范式的根本区别是什么？
4. 它的学术根源是什么？
5. 它有哪些主要技术路线？
6. 它当前产业落地最成熟的场景是什么？
7. 它有哪些代表玩家？
8. 它最大的风险是什么？
9. 它未来可能如何演进？
10. 如果你要写一篇文章，最好的切入点是什么？

## 17. 参考资料与信息来源

必须列出所有重要引用来源：

1. [标题](URL) — 来源类型：官方文档 / 论文 / GitHub / 公司博客 / 新闻 / 报告；简要说明该来源提供了什么信息。
```

## PDF 生成要求

生成 PDF 时应注意：

- 使用清晰标题层级。
- 表格应尽量保持可读。
- 如果内容较长，应生成目录。
- 中文内容避免乱码。
- PDF 文件名必须清楚标识主题。
- 如果 PDF 生成失败，保留 Markdown 源稿并说明失败原因。

可使用的常见方式：

- Markdown → HTML → PDF
- Markdown → Pandoc → PDF
- Python 脚本生成 PDF
- 浏览器打印 HTML 为 PDF

根据当前环境选择最稳定的方法。

## 写作风格

- 使用中文输出，除非用户要求英文。
- 信息密度高，不写空泛套话。
- 解释复杂概念时先讲人话，再讲专业术语。
- 多用表格做对比。
- 保持研究报告语气，避免营销腔。
- 对不确定信息要说明不确定性。
- 不要编造不存在的论文、公司、产品、数据或链接。
